# Pebble — техническая архитектура и roadmap для IT

## Текущий стек (as-is)

| Слой                        | Технология                                                                                                             |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| Backend                         | Python / FastAPI (async)                                                                                                         |
| Frontend                        | React 18 + TypeScript, MUI, AG Grid, Vite                                                                                        |
| БД                            | SQLite (WAL mode, async через aiosqlite)                                                                                    |
| Аутентификация    | JWT + bcrypt                                                                                                                     |
| Деплой                    | Docker → Render.com (persistent disk)                                                                                           |
| AI                              | Anthropic Claude API (импорт Excel, чат, формулы)                                                                |
| Расчётный движок | Собственный: токенизатор + рекурсивный парсер + lazy evaluation с мемоизацией |

## 1. Безопасность

### Что есть сейчас

- JWT-аутентификация (HS256, expiry 7 дней)
- Пароли хешируются bcrypt
- Row-level permissions (права на уровне строк аналитик)
- Роли: пользователь / администратор

### Что реализуем под требования клиента

| Требование                                | Решение                                                                                                                                                                                                             | Сложность   |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------- |
| **MFA / 2FA**                                 | TOTP (Google Authenticator, Microsoft Authenticator) — библиотека `pyotp`, 1-2 дня работы. Либо интеграция с корпоративным SSO                                           | Низкая         |
| **SSO / LDAP / AD**                           | SAML 2.0 или OpenID Connect через `python-saml` / `authlib`. Пользователи логинятся корпоративным аккаунтом, Pebble не хранит пароли вообще     | Средняя       |
| **Шифрование данных at rest** | SQLite → PostgreSQL с TDE (Transparent Data Encryption), либо шифрование тома на уровне ОС (LUKS / BitLocker / AWS EBS encryption)                                                           | Низкая         |
| **Шифрование in transit**           | TLS 1.3 — уже есть на Render; при on-prem деплое — nginx/Caddy reverse proxy с сертификатом                                                                                               | Низкая         |
| **Audit log**                                 | Таблица `audit_log(user_id, action, entity, timestamp, ip, details)` — логирование всех действий: кто, что, когда изменил                                               | Низкая         |
| **Session management**                        | Ограничение одновременных сессий, автоматический logout по таймауту бездействия, revocation list для JWT                                               | Низкая         |
| **IP whitelisting**                           | Middleware: разрешить доступ только с определённых IP/подсетей                                                                                                                   | Низкая         |
| **Row-level security**                        | Уже реализовано — пользователь видит только свои подразделения/регионы                                                                                       | Есть             |
| **CORS policy**                               | Сейчас `*` (открыто) — при продакшене ограничим конкретным доменом                                                                                                  | Тривиально |
| **Penetration testing**                       | Готовы к внешнему пентесту; OWASP Top 10 покрыт архитектурно (параметризованные запросы, нет SQL injection, XSS фильтрация через React) | —                   |

### Вариант "внутренняя сеть"

Если приложение разворачивается **в периметре корпоративной сети** (on-prem / private cloud), многие вопросы снимаются автоматически: нет публичного доступа, трафик не покидает сеть, аутентификация через корпоративный AD/SSO. В этом случае Pebble — просто ещё один внутренний сервис за корпоративным firewall.

## 2. Производительность

### Масштаб задачи

```
Базовая модель:    ~50 000 ячеек (индикаторы × периоды × листы)
+ 10 000 подразделений × 10 версий = 100 000 комбинаций аналитик
Итого:             50 000 × 100 000 = 5 000 000 000 (5 млрд) ячеек — worst case
Реалистичный кейс: ~500M ячеек (не все индикаторы размножаются по всем аналитикам)
```

### Текущий движок

- Однопоточный Python, lazy evaluation с мемоизацией
- На модели 50K ячеек — работает мгновенно
- На 500M ячеек — не справится в текущем виде

### План масштабирования (по уровням)

#### Уровень 1: Оптимизация Python (до ~5M ячеек)

- **Numpy/Pandas vectorization** — формулы типа `SUM`, `AVERAGE`, арифметика пересчитываются векторно, не по ячейке
- **Topological sort + batch eval** — построить DAG зависимостей, считать слоями (все ячейки без зависимостей → все ячейки зависящие только от уже посчитанных → ...)
- **Incremental recalc** — при изменении одной ячейки пересчитываем только зависимый подграф, а не всю модель
- Ожидаемое ускорение: **10-50x**

#### Уровень 2: Rust-движок (до 500M+ ячеек)

- **Rust calculation core** вызываемый из Python через `pyo3` (PyO3) — бесшовная интеграция с FastAPI
- **Всё в памяти** — DAG зависимостей, значения, формулы загружаются в RAM при старте, расчёт без обращений к диску
- **Sparse storage** — хранить только непустые ячейки, формулы разворачивать лениво
- **Автопараллелизация** — Rayon (Rust) автоматически распределяет независимые ветви DAG по всем ядрам CPU
- SIMD-оптимизации для массовых арифметических операций
- **Incremental recalc** — при изменении одной ячейки пересчитывается только зависимый подграф, а не вся модель

**Требования к серверу для 100M ячеек:**

| Параметр                         | Минимум                        | Комфортно |
| ---------------------------------------- | ------------------------------------- | ------------------ |
| RAM                                      | 16 GB                                 | 32 GB              |
| CPU                                      | 8 cores                               | 16 cores           |
| Диск                                 | SSD (для персистенции) | SSD                |
| Время полного расчёта | ~5-15 сек                          | ~1-3 сек        |
| Incremental recalc                       | < 100 мс                            | < 50 мс          |

> Для сравнения: на ноутбуке Apple M2/M3 (16 GB unified memory, 8-12 ядер) 100M ячеек рассчитываются за 1-3 секунды. На продакшен-сервере — быстрее.
> Стоимость сервера: Hetzner ~€50/мес, AWS r6i.xlarge ~$150/мес. Никакого спецжелеза или кластера.

### База данных: роль и выбор

С Rust-движком БД — это **persistence layer**, а не вычислительный слой:

```
Старт / открытие модели:  БД → загрузка в RAM → Rust-движок
Расчёт:                   Полностью в RAM, БД не участвует
Сохранение:               Rust → batch write в БД (асинхронно, в фоне)
```

#### SQLite → PostgreSQL при масштабировании

| Аспект                          | SQLite (сейчас)                                                                                         | PostgreSQL (при масштабировании)                                             |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Скорость расчёта       | Одинаково — расчёт в RAM                                                                     | Одинаково — расчёт в RAM                                                      |
| Конкурентная запись | 1 writer, блокирует                                                                                  | Полная конкурентность                                                      |
| Масштаб данных           | до ~10 GB комфортно                                                                                | Терабайты без проблем                                                       |
| 500M ячеек (~50 GB)              | Работает, но некомфортно (бекап/vacuum медленные, нет партиций) | Рутина. Partitioning по моделям,`DROP PARTITION` мгновенно           |
| Batch write 100M ячеек           | ~10-30 сек (один writer)                                                                               | ~3-5 сек (параллельные writers)                                                 |
| Бекапы                          | Копия файла / Litestream                                                                            | pg_dump (параллельный), WAL archiving, point-in-time recovery                      |
| Миграция                      | —                                                                                                            | 1-2 дня: заменяем `aiosqlite` → `asyncpg`, схема 1:1 совместима |

#### Почему не ClickHouse / TimescaleDB / DuckDB?

| БД        | Для чего создана                                                         | Почему не нужна                                                                                |
| ----------- | -------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| ClickHouse  | Аналитика на миллиардах строк (GROUP BY, агрегации) | Мы не агрегируем в БД — загружаем в RAM и считаем в Rust               |
| TimescaleDB | Временные ряды (IoT, метрики)                                      | Наши периоды — не time series в классическом смысле                        |
| DuckDB      | "SQLite для аналитики", columnar                                           | Интересен, но нет конкурентности — та же проблема что у SQLite |

**PostgreSQL — оптимальный выбор.** 50 GB для Postgres — мелочь. Весь расчёт в памяти, БД только хранит и отдаёт данные. На этой роли Postgres справится и с 5 млрд ячеек.

**Важно:** архитектура приложения уже готова к замене БД — весь SQL изолирован в `db.py`, замена драйвера не затрагивает бизнес-логику.

### Стратегия для UI при больших моделях

- **Виртуализация таблицы** — AG Grid уже поддерживает virtualized rows/columns, рендерит только видимую область
- **Серверная пагинация** — загружать данные порциями (по листу, по аналитике)
- **Web Workers** — тяжёлые UI-операции (сортировка, фильтрация) в отдельном потоке, UI не блокируется

## 3. Прочие требования IT

### Инфраструктура и эксплуатация

| Тема                         | Текущее состояние | Production-ready решение                                                                                                                                                      |
| -------------------------------- | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Бекапы**           | Копия SQLite файла      | Автоматические ежедневные бекапы: pg_dump (PostgreSQL) или Litestream (SQLite → S3). Point-in-time recovery. Retention policy (30/90/365 дней) |
| **Disaster Recovery**      | —                                | RPO < 1 час, RTO < 4 часа. Реплика в другом регионе. Documented runbook для восстановления                                              |
| **High Availability**      | Один инстанс           | Несколько инстансов за load balancer (nginx/HAProxy). Health checks уже есть (`/api/models`). Auto-restart при падении                        |
| **Мониторинг**   | —                                | Prometheus + Grafana: CPU, RAM, response time, error rate, active users. Алерты в Slack/email при аномалиях                                                       |
| **Логирование** | stdout                            | Structured logging (JSON) → ELK stack (Elasticsearch + Kibana) или Loki + Grafana. Retention 90 дней                                                                         |
| **CI/CD**                  | Push → Render auto-deploy        | GitHub Actions: lint → test → build → deploy staging → smoke test → deploy prod. Rollback в один клик                                                                  |
| **Environments**           | Один (prod)                   | dev → staging → production. Staging — точная копия prod для тестирования                                                                                |

### Управление данными

| Тема                                  | Решение                                                                                                                                                             |
| ----------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Data retention / очистка** | Политика хранения: архивация старых моделей, автоудаление через N месяцев по желанию клиента |
| **Экспорт данных**     | Уже есть экспорт в Excel. Можно добавить CSV, PDF, API для интеграций                                                             |
| **Импорт данных**       | Уже есть интеллектуальный импорт Excel через AI (Claude)                                                                                 |
| **Multi-tenancy**                   | Варианты: schema-per-tenant (PostgreSQL), отдельная БД на клиента, или row-level isolation. Зависит от требований        |
| **Data residency**                  | Деплой в нужном регионе / стране. Docker-образ разворачивается где угодно: AWS, Azure, GCP, on-prem                 |

### Интеграции

| Тема                       | Решение                                                                                                                                                                                                  |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **API**                  | REST API уже есть (FastAPI автогенерирует OpenAPI/Swagger документацию по адресу `/docs`). Готов к интеграции с внешними системами |
| **Webhooks**             | При изменении данных — POST на указанный URL. Реализуется за 1 день                                                                                              |
| **SSO / Directory Sync** | SAML, OIDC, SCIM — стандартные протоколы, библиотеки существуют                                                                                                        |

### Compliance и governance

| Тема                      | Решение                                                                                                                                                                       |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Audit trail**         | Полный лог действий: кто, когда, что изменил, старое и новое значение                                                         |
| **Version history**     | История изменений каждой ячейки с возможностью отката                                                                                 |
| **Access control**      | RBAC (роли: viewer, editor, admin) + row-level permissions по аналитикам                                                                                             |
| **Data classification** | Метки конфиденциальности на уровне модели/листа                                                                                            |
| **GDPR / ПДн**       | Если в моделях нет персональных данных — не применяется. Если есть — реализуем right to erasure, consent management |

## Ключевое сообщение для IT

> **Pebble — это MVP с production-grade архитектурой.** Стек (FastAPI + React + Docker) — это стандартные, зрелые технологии с огромным комьюнити. Архитектура модульная: каждый компонент (БД, аутентификация, расчётный движок) заменяется независимо, без переписывания системы. Все перечисленные улучшения — это добавление слоёв поверх существующей архитектуры, а не переписывание с нуля.
