# План: Полный UI E2E тест (Playwright)

Цель: выполнить полный сценарий работы с моделью только через UI —
от чистой БД до проверки консолидации. Никаких прямых API-вызовов
для действий, только навигация и клики.

## Архитектура тестов

- Файл: `tests/test_full_ui_flow.py`
- Фреймворк: pytest + playwright (sync API, уже установлен)
- Run: `pytest tests/test_full_ui_flow.py -v -s --headed` (увидеть браузер)
- Headless: `pytest tests/test_full_ui_flow.py -v -s`
- Таймаут: большой, ~10 мин на импорт Excel

Тесты идут последовательно в одной module-scope сессии браузера,
каждый шаг — отдельный `test_NN_*` для удобной отладки.

## Предусловия

1. Backend бежит на :8000 (или :3000 — проверить). `lsof` показывает
   :8000 backend (uvicorn) и :3000 node (vite dev).
2. Файл `models.xlsx` в корне проекта (~/pebble/models.xlsx) — эталон.
3. Пользователь `admin/admin` — создаётся автоматически в `db.py:init_db`.

## Шаги

### Шаг 0. Reset БД

- Удалить `pebble.db`, `pebble.db-shm`, `pebble.db-wal`.
- Рестарт backend: убить uvicorn, запустить `python start.py` в фоне.
- Подождать, пока /health отвечает 200.
- Защита: если в БД уже есть модель VERIFIED и нет analytic "dept",
  допустимо начать с неё (идемпотентность). Но чистый сценарий —
  начинать с нуля.

### Шаг 1. Логин admin/admin

- `page.goto(BASE)` → очистить `localStorage` → перезагрузить
- Заполнить `input[name="username"]="admin"`, `input[name="password"]="admin"`
- Клик `button:has-text("Войти")`
- Ждать: `input[placeholder*="Поиск"]` видим (главный экран)

### Шаг 2. Upload эталонной модели

- Клик по кнопке импорта (`[title="Импорт модели из Excel"]`)
- Заполнить input `[placeholder="Название модели"]` → "VERIFIED"
- `page.locator('input[type="file"]').set_input_files("models.xlsx")`
- Клик `button:has-text("Импортировать")`
- Ждать появления текста типа "Закрыть" или "готово" (до 5 мин)
- Закрыть диалог
- Проверить: `text=VERIFIED` в tree

### Шаг 3. Проверка чисел на эталонной модели

- Развернуть VERIFIED в дереве
- Открыть лист "кредитование" (BaaS.1)
- Снять несколько опорных чисел из DOM (знаем несколько ячеек из
  `test_import_verify.py`). Например: общий выдано кредитов за год.
- Альтернатива: открыть разные листы и убедиться, что в гриде
  есть числовые значения и формулы отрабатывают.
- Ключевая проверка: таблица `<table>` видна, не пустая.

### Шаг 4. Создать аналитику "Подразделение"

- В левом дереве под VERIFIED найти папку "Аналитики" → expand.
- Клик на `[title="Добавить аналитику"]` (или по + рядом с папкой).
- В центре появится AnalyticSettings.
- Заполнить название "Подразделение" (код podrazdelenie авто).
- Добавить корневую запись "Головной" (кнопка `[title="Добавить запись"]`).
- Навести на строку "Головной" → добавить дочернюю "Филиал 1" и "Филиал 2"
  (`[title="Добавить дочерний"]`).
- Сохранить (автосейв на blur, или явно нажать Enter).

### Шаг 5. Добавить аналитику во все листы

- На экране AnalyticSettings виден `button:has-text("Добавить во все листы")`.
- Клик → ждать появления "Добавлено в N лист(ов)" (success caption).
- Проверить, что сообщение видно.

### Шаг 6. Проверка: числа на Ф1 = оригинал, Ф2 пусто, Голова = Ф1

- Открыть любой лист (например, кредитование).
- В верхней панели должен появиться «pin»-селектор аналитики Подразделение.
  Логика app: если аналитика добавлена на лист, она появляется
  в строках или в pin-панели.
- Pin'нуть на "Филиал 1" → числа = оригинал.
- Pin'нуть на "Филиал 2" → все ячейки пустые (мы ничего не вводили).
- Pin'нуть на "Головной" → числа = Ф1 (т.к. Ф2=0).
- Снять 2-3 контрольных числа через `page.locator('td')` и сравнить
  с оригиналом.

### Шаг 7. Создать пользователей dep1, dep2

- Открыть Users dialog (`[title="Пользователи"]` в тулбаре или
  кнопка с иконкой people).
- Клик `[title="Добавить пользователя"]` → появится "Новый пользователь"
  в списке, выбрать его, переименовать поле в "dep1".
- Клик `[title="Сменить пароль"]` → ввести "dep1" → Сохранить.
- Перейти в раздел "Листы": для VERIFIED включить can_view+can_edit для
  всех листов.
- Раздел "Аналитики": развернуть Подразделение → Головной →
  снять галочки с Головной и Ф2, оставить только Ф1 (view+edit).
- Повторить для dep2 (пароль dep2, доступ только к Ф2).
- Закрыть диалог.

### Шаг 8. Logout → login dep1

- Выйти (кнопка logout в тулбаре, обычно иконка logout).
- Логин dep1/dep1.
- Открыть любой лист модели VERIFIED.
- Проверить: auto-pin на Ф1 (ведь ему доступна только одна запись).
- Ввести в редактируемую ячейку число (например, на "параметры модели"
  найти manual-ячейку и ввести 999999). Enter.
- Проверить: ячейка сохранилась, пересчёт прошёл.

### Шаг 9. Logout → login dep2, ввести данные

- Аналогично dep1. Auto-pin на Ф2.
- Ввести в ту же manual-ячейку 111111.
- Проверить сохранение.

### Шаг 10. Logout → login admin, проверить консолидацию

- Войти admin/admin.
- Pin на Головной.
- На той же manual-ячейке должно быть 999999 + 111111 = 1111110
  (или близко к этой сумме, в зависимости от округления).
- Проверить через `expect(cell).to_have_text(...)` с допуском.

## Стратегия отладки

- Каждый шаг — отдельный test_N_*. pytest упадёт на первой
  ошибке, дальше можно запускать только конкретный шаг
  (через `pytest -k test_04_add_analytic`).
- На падении: скриншот `page.screenshot(path=...)` и HTML-дамп.
- Селекторы могут не совпадать — итеративно чинить, смотреть
  в dev-tools через `--headed`.
- Таймауты: upload ~5мин, остальное 30с.
- Если auto-pin не срабатывает — смотреть логику в PivotGrid.tsx
  (строки 417-434) и править тест под реальное поведение.

## Критерии успеха

- Все 10 тестов проходят `pytest -v` без ошибок.
- Финальная проверка: значение на "Головной" = Ф1 + Ф2 с допуском.
- БД оставлена в рабочем виде (модель VERIFIED + dept + dep1/dep2).
