## Нерешённые проблемы (в порядке приоритета)

### 1. Prev-period detection в трансляторе
**Проблема**: В Excel `=D20+E17-E19` (формула m2), D = предыдущий период, E = текущий.
Транслятор не различает — всё пишет как `[name]`.

**Решение**:
- Определить "базовую" колонку формулы (E для m2, D для m1)
- Если ссылка на колонку ЛЕВЕЕ базовой → добавить `(периоды="предыдущий")`
- Если на ту же колонку → текущий период
- Особый случай: m1 формула с ссылкой на C (до первого периода) → заменить на 0

**Файл**: Создать `backend/excel_formula_translator.py`

### 2. OPEX маппинг (3-колоночная структура)
**Проблема**: OPEX имеет данные в колонках A/B/C, не только A. Маппинг 18/127.

**Решение**:
- OPEX rows: col A = Продукт, col B = Вид расхода (OPEX/CAPEX), col C = Статья
- Маппинг: использовать col C для имени показателя, col A для группы
- Compound name: "{Статья} ({Продукт})" — уже так в записях
- Дополнительно: rows без col A — это summary rows (col B = OPEX total)

### 3. Cross-sheet refs к groups без ячеек
**Проблема**: `[OPEX+CAPEX::Потребительский кредит]` → группа без cell data.
Excel `=-'OPEX+CAPEX'!E19` = конкретная строка (Расходы на персонал).

**Решение**:
- НЕ fallback в движке (движок exact)
- Формула должна ссылаться на КОНКРЕТНУЮ запись: `[OPEX+CAPEX::Расходы на персонал (Потребительский кредит)]`
- Транслятор должен использовать row→record mapping OPEX листа для правильного имени

### 4. Применить транслятор ко всем формулам
**Шаги**:
1. Для каждого листа и каждой formula cell:
   - Получить Excel формулы (m1=col[data_start], m2=col[data_start+1])
   - Перевести через транслятор
   - Записать в cell_data.formula
2. Пересчитать всю модель: `calculate_model(db, mid)`
3. Сверить с Excel data_only значениями
4. Итерационно исправлять

### 5. Верификация и сохранение эталона

- Цель: 100% совпадение (кроме OPEX input cells где формулы в жёлтых ячейках)
- Когда 100% — сохранить модель как "BaaS (эталон)"
- Потом улучшать Claude prompt до автоматического 100%
